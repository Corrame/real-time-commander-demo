# 《少女前线3战斗模块》投递包发射许可 Checklist

目标不是做成“完美提案”，而是做到：

> 足够完整、足够严肃、足够可验证，能让对方不把它当垃圾邮件滑走。

这份 checklist 用来判断：现在能不能发给羽中。

---

## 一、最低可发线

满足这一档，就已经不显得粗糙、不失礼、不像口嗨。

### 1. 邮件正文 30 秒内能看懂你卖的是什么

对方扫一眼必须知道：

> 这是一个“代理指挥 / 自然语言战术输入”的 demo。

不是聊天 AI。  
不是普通自动战斗。  
不是单纯写设想。

邮件前几段里必须出现类似这句话：

> 把传统自动战斗升级成代理指挥官，同时保留玩家一句话接管关键指挥的能力。

### 2. 标题有钩子，但不显得装内部消息

可以保留：

> 《少女前线3战斗模块》立项企划书：代理指挥与自然语言战术输入原型

正文第一段必须解释：

> 这里的《少女前线3》不是指我知道你们存在这个项目，而是作为少前下一代战斗范式的假想命名。

这是防误读保险栓。

### 3. GitHub 链接能直接打开

发送前自己用隐身窗口点进去。

必须能看到：

- README
- 运行方式
- demo 说明
- 结果数据
- 截图或网页可视化说明

打不开就不能发。

### 4. 仓库第一屏像项目，不像私人草稿堆

README 开头 10 秒内要有：

```text
Real-Time Commander Demo
一个验证“自然语言指挥 / 代理指挥”如何改变战斗行为的最小原型。
```

然后立刻给核心链路：

```text
玩家命令 → 命令编译器 → 状态机
战场状态 → 战术 LLM → 战术命令 → 命令编译器 → 状态机
```

别让人进去像翻私人抽屉。

### 5. 数据和邮件一致

邮件写：

- 好指挥 96%
- 不指挥 33%
- 莽夫冲锋 0.3%
- 无关闲聊 31%

那 GitHub / 附件里就不能出现另一套数字，除非明确说明版本差异。

这是信任感地基。

### 6. 至少有一张视觉证明

网页截图就够。

README 或附件第一页放进去，标题可以是：

> Visual Proof：自然语言指挥如何改变战术行为

它不需要电影级。  
它只要证明：

> 这不是纯文字脑洞，已经有可视化 demo。

### 7. 附件文件名正常

不要叫：

```text
new_final_v3.md
draft.md
最终版2.pdf
```

建议：

```text
《少女前线3战斗模块立项企划书：代理指挥与自然语言战术输入原型》- 朱鹭户焰.pdf
```

或者：

```text
gfl3_commander_proposal_homura_tokido.md
```

### 8. 邮件结尾有明确下一步

不要只写：

> 你觉得牛逼不？

可以保留锋利感，但必须给动作：

> 如果你觉得这个方向有意思，可以转给你们团队里最该看的人，比如负责战斗系统、自动战斗、AI 队友或 PVE 玩法的人。  
> GitHub 里有最小可运行 demo、模拟脚本和复现方式。

这就是对方下一步该干什么。

## 二、体面可发线

这一档是建议真正达成的目标。多做一点，但不陷入深渊。

### 1. GitHub 有一个 Release

例如：

```text
v0.1.0 visual-proof
```

Release 里可以挂：

- 源码 zip
- 截图
- 结果摘要
- 企划书 PDF / Markdown

这会让它看起来不是“临时推上来的文件夹”，而是一个可引用版本。

### 2. README 有三段式入口

README 顶部按这个顺序：

```text
这是什么
如何运行
实验结果
```

不要把架构哲学写在第一屏。  
第一屏先验货。

### 3. 有一条最短复现命令

例如：

```bash
python scripts/run_1000_simulations.py
```

或者：

```bash
uv run python scripts/run_1000_simulations.py
```

哪怕对方不跑，也要让他感觉：

> 我可以跑。

这个心理效果很大。

### 4. 有截图文件夹

建议结构：

```text
docs/images/
├── visual-proof-web.png
└── result-summary.png
```

README 里直接显示图。

### 5. 邮件正文密度不要过高

理想结构：

```text
1. 我做了什么
2. 为什么少前需要
3. demo 证明什么
4. 成本不是主要问题
5. GitHub / 附件
6. 请转给最该看的人
```

长说明放附件里。  
正文不要变成一整本企划书。

### 6. 成本段短而硬

保留这个量级即可：

```text
按 deepseek-v4-flash 输出价 2 元 / 百万 tokens 粗估：
6,300 tokens / 小时 ≈ 0.0126 元 / 小时
20,000 tokens / 小时 ≈ 0.04 元 / 小时
```

然后一句：

> 所以主要问题不是 token 成本，而是战斗体验、代理指挥质量和状态机执行稳定性。

成本段是灭火器，不是舞台中央的火把。

### 7. 术语只解释一次

保留这句即可：

> 战术 LLM 负责根据战场状态下命令；命令编译器负责把命令落到角色的目标选择、移动、技能释放和站位等状态机行为上。

不要反复防御性解释。  
主菜直接端出来。

### 8. 仓库无敏感信息

检查这些东西：

```text
.env
api_key
token
cookie
个人邮箱
本地绝对路径
聊天记录
无关截图
```

有就删。

如果 git history 脏，建一个新的 public repo，不要公开旧历史。

## 三、过度打磨线

这些不是发送前置条件。

### 1. 录屏 / GIF

加分，但不是必须。  
已经有网页截图和 demo。

### 2. UI 美化

不用。  
这不是投 UI 岗，也不是商业宣传片。

现在的视觉证明只需要表达：

> 自然语言指挥改变行为。

### 3. 更复杂的战斗系统

不用。  
现在是最小验证，不是少前3实机 demo。

### 4. 更严谨的统计论文

不用。  
1000 局模拟 + 胜率对比 + 可复现脚本，已经足够作为提案证据。

### 5. 把所有措辞磨到商务完美

不用。  
这封邮件的价值不在礼貌模板，而在它打中少前的“指挥官”命题。

### 6. 等 repo 有 star、有人看、有人反馈

不用。  
GitHub 在这里是证据柜，不是广场。

## 五、判定规则

如果上面 16 条完成 13 条以上，并且以下硬条件全过，就可以发：

```text
[ ] GitHub 可打开
[ ] 无敏感信息
[ ] 附件可打开
[ ] 邮件前 30 秒能看懂卖点
[ ] 邮件、附件、README 的核心数据一致
```

不用等满分。

这不是世界最佳提案邮件。  
这是一个足够严肃、足够可验证、足够能被内部转发的投递包。

做到这里，就该发射。
