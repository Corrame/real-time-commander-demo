# 立项答疑：羽中预期质疑 × 现有代码实际能证什么

> 回答原则：**只从"现有代码实际证明了什么"出发**，不做愿景式辩护。知道自己证明了什么、没证明什么，本身是主策划素质。
>
> 三档诚实标注：
> - **【实证】** 代码已实现 / 有运行数据 / 能正面硬刚
> - **【架构支持·未落地】** 设计已想清、结构上支持，代码尚未实现
> - **【代码解决不了】** 需真人验证 / 需进厂资源 / 属商业决策——诚实认边界

---

## 核心架构：我们跑过的不是一个 demo，是两个独立模拟，各证因果链的一半

完整命题是一条三段链：

```text
自然语言  ──①──>  结构化命令  ──②──>  胜率变化
```

这两段的实验要求是**互相冲突**的，所以必须用两个独立模拟分别证明——这是受控实验的正确做法，不是将就：

| | 老模拟（遭遇战，archive/legacy_encounter/） | 新模拟（镜像 3v3，game/mirror_map_sim.py） |
|---|---|---|
| 证哪一段 | ① 自然语言 → 结构化命令 | ② 结构化命令 → 胜率变化 |
| 为什么必须是它 | 要证 LLM 能解析"多单位、带条件/约束/优先级"的复杂指挥，**场景必须丰富、不对称**（4 单位 vs 装甲/无人机，有破盾、隐蔽、处决时机） | 要让胜率倾斜可信地归因到输入，**双方必须完全对称、固定种子** |
| 代码证据 | `command_interpreter.py` 把一句中文解析成多个 `Order(unit, action, target, condition, constraints, priority)`；`ALLOWED_ACTIONS/POLICIES/TARGETS` 白名单 | 6 policy / 9 mode 的 `UnitCommand`；固定种子逐字节可复现；镜像对称五五开（已验） |

一个要"丰富不对称"，一个要"极简对称"，需求相反，**不可能塞进同一个 sim 而不把证据搅浑**。

**1.0 已搭好的桥：** `nl_command_eval.py` 已端到端跑通"自然语言 → LLM → 6 个 FSM policy → 胜率"，本会话现场真调验证过。它是粗粒度（6 policy）的连通证明。

**roadmap（1.1，未做）：** 把老模拟那套**富结构化命令**接到新模拟的对称测试台，端到端用 per-unit order 跑胜率。零件都在，是工作量问题，留待进厂。

## A. 玩家体验

### A1. 玩家真愿意打字/说话吗？打字算不算另一种麻烦？
**【架构支持·未落地】+ 体验部分【代码解决不了】**

- 设计原则"**入口宽、执行窄**"：自然语言只是加速搜索层，底层是一组常驻菜单可见的有限按钮。懒得说话→点固定菜单；想一句顶十个操作→才打字/语音。两条路通向同一组有限命令。
- 代码现状：解析层已能把文本映射成确定命令（老模拟的 `Order` / 新模拟的 policy）；浮动按钮 UI、常驻菜单、语音输入**前端未实现**。
- 诚实缺口：玩家到底愿不愿意打字是体验问题，**代码证明不了，要真人测**。能保证的只是：不打字也能玩。

### A2. 新手怎么知道说什么有用？自由输入的边界焦虑？
**【实证（机制层）】+【架构支持·未落地（引导层）】**

- 执行空间**有限且可枚举**，边界客观存在：老模拟有 `ALLOWED_ACTIONS / ALLOWED_TARGETS`，新模拟有 6 policy + 9 mode。菜单把所有可能动作列出，自由输入只是快捷方式——不是无底洞。
- 诚实缺口：把这条边界做成新手一眼看懂的 UI 引导，**未实现**。

### A3. 第 50 次刷同一关，会退化成复制粘贴 / 看动画吗？
**【架构支持·未落地】**

- 退化成"看动画/挂机"在设计里**是预期内的正确行为**：系统能秒算结果，碾压局分诊放行，玩家直接跳过，注意力只花在均势局。
- 诚实缺口：战前胜率预判 / 分诊放行**未实现**，目前是设计。命门是"分诊置信度"的准度。

## B. 机制深度

### B1. 只有 6 个策略桶，自然语言是不是噱头？决策空间真实大小？
**【实证（老模拟）】+ 诚实标注：产生胜率证据的那条链目前是 6-policy 粒度**

- 决策空间**不止 6**。6 是新模拟里的队伍方针层。**老模拟已实证**真实空间是组合乘积：`action(~10) × target × condition × constraints × priority × 每个单位独立下令`。一条命令"MG 压制中路/RF 打无人机/SMG 左翼隐蔽/HG 等破盾处决"被解析成 **4 个单位各自不同的结构化 order**（`command_interpreter.py` 的解析输出即证据）。
- **诚实分界**：这套丰富度在**老模拟**里证过；而产生 96% 那组胜率证据的**新模拟**，当前只用到 6 policy 这一粗粒度。"自然语言→富命令→胜率"的端到端（1.1）还没合龙——但两半边各自都是实证，不是空话。

### B2. PvE 关卡固定，最优指挥被攻略固化，会不会掉回《圣兽之王》那个坑？
**【实证（半）】+【代码解决不了（半）】**

- 能答一半：圣兽之王栽在"深度没回报"。新模拟的"**命令必须改变胜率**"已实测——好指挥 96.3% / 莽夫 0.3% / 不指挥 ~33%，且**对手越强指挥越被需要**（见 1.0_EVIDENCE_ROBUSTNESS.md：换成会打的对手后，不指挥 0% vs 好指挥 23.7%）。指挥**有回报**，这条立得住。
- 诚实缺口：最优解被攻略固化是另一问题，**代码解决不了**，需关卡随机/动态对手/阶段难度，属进厂内容侧。

### B3. 镜像 3v3 是玩具规模。scale 到多兵种/技能/buff/地形，还干净吗？
**【架构支持·未落地】+【代码解决不了（scale 验证）】**

- 当前只验证小规模，**代码不能证明 scale 后仍干净**，诚实开放问题。
- 架构理由："窄而正交的动作集"让 scale 增加的是 target/action 种类，路由难度**线性**上升而非组合爆炸；归因干净性靠"确定性引擎+固定种子"，规模变大原理不变（已实测逐字节可复现）。
- 诚实缺口：原理支持，实测没做。

## C. 工程与成本

### C1. 每场调一次 LLM，延迟多少？弱网？
**【实证（有数据）】+【架构支持·未落地（消化机制）】**

- 实测：当前重推理配置（high reasoning + thinking，见 `llm_client.py`）单次解析约 **25.8 秒**（太重）；裸调用约 **1.5 秒**。延迟主要来自推理配置选型，不是链路本身。
- 消化路径：双通道（按钮即时、语言异步）+ 碾压局免调（分诊）+ 轻量配置调优。
- 诚实缺口：分诊免调、轻量配置属未落地优化；但延迟有明确消化路径，非死结。

### C2. 百万 DAU × 每战一次 LLM，成本量级？能压下来吗？
**【架构支持·未落地】**

- 关键红利：**执行空间有限 ⇒ 可缓存、可预编译**。常见指令（"集火残血""撤回保人"）落到可枚举有限动作集，完全可缓存命中，不必每次调 LLM。
- 诚实缺口：成本测算、本地小模型、缓存层均**未实现**；但"窄执行"天然给了降本结构。

### C3. 同一句话两次给不同策略，玩家会当 bug。确定性怎么保证？
**【实证·强项】**

- 确定性不锚在 LLM 输出，锚在代码事实：① 白名单清洗（老模拟 `_clean_action/_target/_policy` 落到封闭集；新模拟 `clean_policy_result` 落到 6 policy）；② 映射成可见按钮→玩家确认→确定性引擎执行；③ 引擎本身固定种子**逐字节可复现**（已实测），temperature=0.2 进一步降随机。
- 即便 LLM 措辞解析略浮动，落点仍是同一组有限按钮——玩家感知到的是确定按钮，不是 bug。

### C4. prompt injection 骗超强策略 / 违规输入怎么防？
**【实证·强项】**

- "**LLM 关在出错无代价的笼子里**"，代码**已实现**：策略空间是封闭白名单（老模拟 `ALLOWED_ACTIONS/POLICIES/TARGETS`，新模拟 6 policy）。LLM 哪怕被注入吐"无敌""秒杀"，不在白名单的动作**直接被丢弃**——造不出不存在的合法动作。违规内容只影响"映射到哪个合法动作"，碰不到执行与数值。结构性安全，不是补丁。

> 注：1.0 主路径**故意不做无 LLM 的本地 fallback**——这是设计决定，不是缺口。无 LLM 的指挥对本项目没有意义；解析失败就失败，不伪造一个假命令糊弄玩家。（老模拟的 `_fallback_interpret` 是旧路径应急产物，非当前主张。）

## D. 商业与 IP 咬合（命门）

### D1. 替换现有战斗，还是新增模式？
**【代码解决不了（商业决策）】**

- 代码无关，产品决策。能提供的论据：少前 2 战斗系统是**公认短板**（玩家批"古典/冗长/缺策略"，官方承认战棋玩法"已不适应市场"，公测后流水断崖）。换战斗心脏的正当性来自**市场与官方双重背书的数据**，非主观判断。

### D2. 跟养成/数值/gacha 怎么咬合？指挥决定胜负的话，氪金强角色还重要吗？
**【实证（结构层）】+【论述（咬合逻辑）】**

- 指挥**不决定胜负**。三层栈：战略（养成/编队/资源）定大盘期望胜率 > 战术（语言指挥）只在均势局撬动 > 执行。强角色在碾压局大杀四方（养成兑现），指挥只在均势局起作用。
- **指挥放大养成价值，不替代它**：激励逻辑是"夸高练度角色→加成"，养成越深可激励的角色越强，二者是乘法不是替代，付费动机反被强化。
- 代码层：老模拟里 `commander_policy`（指挥层）与单位养成数值（morale/accuracy 等）在引擎里**分离且叠加**，结构上支持"养成定底、指挥微调"。

## E. 证据的边界（懂数据的人会问）—— 最考验诚实

### E1. 你证明了"胜率会变"，但"会变 ≠ 好玩"。有真人测过吗？
**【代码解决不了·诚实认输】**

- **没有真人测过。代码证明不了好玩。** 这是当前**最大的未证明项**，不掩盖。对比演示只能证"命令可复现地改变战局"，不能证"任何人觉得它有趣"。下一步唯一真正算数的动作，是拿能感受的最小原型找真人玩五分钟。在那之前，"好玩"都是逻辑自洽，不是事实。

### E2. good_focus 赢 96%，是客观更优，还是 dumb 基线太蠢？怎么排除自导自演？
**【实证（强化版）】+【代码解决不了（绝对最优性）】**

- 能排除大半：① **镜像对称五五开已验**——双方同策略时引擎产出约 50/50（good_focus 对 good_focus 实测 23.7/23.7），证明引擎不偏袒任一方。② **决策意义不依赖弱对手**——本会话补测：把对手从木桩换成会打的 good_focus 后，不指挥被打到 0%、好指挥仍能打成 23.7%（见 1.0_EVIDENCE_ROBUSTNESS.md）。对手越强指挥越关键，与"靠蠢对手撑场"正好相反。
- 诚实缺口：96% 这个**绝对数字**仍部分取决于 baseline 设定。代码能证"对称公平 + 决策有后果 + 不依赖弱对手"，**不能证"这个策略是客观全局最优"**。要钉死绝对值需更强基线或真人对局校准。

## 总结：这套代码的诚实定位

| 能正面硬刚（实证） | 架构支持·未落地 | 代码解决不了 |
|---|---|---|
| B1 决策空间组合乘积（老模拟） | A1 双通道非强制打字 | E1 好不好玩（要真人） |
| C3 确定性（种子+白名单+确认） | A3 分诊放行 | E2 绝对最优性（方法论边界） |
| C4 注入防御（白名单笼子） | B3 scale 后干净性 | B2 攻略固化（要内容侧） |
| E2 对称公平 + 决策不依赖弱对手 | C1/C2 延迟与成本消化 | D1 替换/新增（商业决策） |
| D2 养成与指挥分层叠加（老模拟结构） | | |

**一句话定位**：两个独立模拟分别证明了"自然语言→结构化命令"和"结构化命令→胜率变化"两个半边，1.0 已用 6-policy 粒度把它们端到端连通。这套代码能扎实证明"机制可落地、执行可控、安全可防、决策有深度、与养成不冲突、且决策意义不依赖弱对手"。它**不**假装能证"商业上好玩、规模上成立、策略绝对最优"——那些是真人测试与进厂资源的事。

**对羽中的真正价值**：不是"这游戏能赚钱"的证据，是"这个人想清楚了边界、知道什么已证明什么没证明、且能把玩法洞察落地成可控系统"的证据——这正是主策划区别于纸上谈兵者的地方。
